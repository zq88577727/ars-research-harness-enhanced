# Artifact Governance

The repository keeps enough artifacts to prove the workflow, but it should not
become a storage bucket for every generated file.

## Commit to the Repository

- Source code, validators, templates, and workflow documentation.
- Small public-use teaching data needed for the canonical NHANES example.
- Curated example outputs that make the S0-S9 loop inspectable.
- Source-backed registries: claims, references, reporting checklist mappings,
  journal profiles, revision traces, and data source manifests.

## Prefer Release Artifacts

- Large DOCX/PDF submission packages.
- High-resolution figures or presentation images.
- Frozen workshop bundles.
- Generated outputs that are useful for readers but not required by validators.

## Regenerate Locally

- Intermediate analysis tables that can be reproduced from scripts.
- Temporary manuscript drafts outside the curated example package.
- Exploratory plots.
- Local CHARLS or GBD derived files that depend on restricted or user-specific
  exports.

## Never Commit

- Restricted raw CHARLS files.
- Private GBD exports when redistribution is not clearly allowed.
- Credentials, cookies, API tokens, or private portal screenshots.
- Participant-level identifiers or re-identifiable records.

## Golden Example Rule

The NHANES example may keep one curated set of final tables, figures, and DOCX
outputs because it demonstrates the end-to-end workflow. Additional generated
packages should be moved to releases or regenerated by scripts unless they are
needed by a validator.
